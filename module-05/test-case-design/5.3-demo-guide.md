## Scene 4 — Test Case Design

**เป้าหมาย:** ให้ผู้ชมเห็นว่า AI ช่วยออกแบบ Test Case ได้ครอบคลุมกว่าคิดเอง และแยก Unit Test ออกจาก Integration/E2E ชัดเจน

| # | ทำบนหน้าจอ | พูด |
|---|-----------|-----|
| 1 | เปิด Terminal → รัน `claude` เพื่อเปิด Claude Code | — |
| 2 | Copy prompt จาก `01-test-case-prompt.md` → Paste → Enter | "เริ่มออกแบบ Test Case ของ Unit Test ก่อนครับ" |
| 3 | รอ response → เลื่อนดู Test Cases ที่ได้ | — |
| 4 | **ชี้ที่ Pass Scenarios (TC-P01–P04)** | "Pass Scenarios — score ≥ 60, interval เพิ่มตาม consecutive: ครั้งแรก→1, ครั้งที่สอง→6, ครั้งต่อไป→คูณ EF" |
| 5 | **ชี้ที่ Edge Cases (TC-E01–E04)** | "Edge Cases — boundary ที่ score 59 กับ 60, และ easeFactor ที่ต้อง clamp ไว้ที่ 1.3 เสมอ" |
| 6 | **ชี้ที่ Fail Scenarios (TC-N01–N02)** | "Fail Scenarios — score < 60 ต้อง reset ทันที: interval กลับ 1, consecutiveCorrect กลับ 0" |
| 7 | **ชี้ที่โครงสร้างของแต่ละ Case** | "แต่ละ Case ระบุชื่อที่สื่อความหมาย — Input SM2Input — และผลที่คาดหวัง" |
| 8 | หยุดนิ่ง 2 วินาที | "Unit Test ทดสอบฟังก์ชันเดี่ยวๆ — ไม่ใช่ทดสอบ flow ทั้งหมดครับ" |

## Scene 5 — Test Data Design

**เป้าหมาย:** ให้ผู้ชมเห็นว่า Test Data ที่ดีต้องระบุ Input และ Expected Output ชัดเจนสำหรับแต่ละ Case

| # | ทำบนหน้าจอ | พูด |
|---|-----------|-----|
| 1 | **ใน Claude Code conversation เดิม** ต่อเนื่องจาก Scene 4 | — |
| 2 | Copy prompt จาก `02-test-data-prompt.md` → Paste → Enter | "ต่อด้วย Test Data สำหรับแต่ละ Case ครับ" |
| 3 | รอ response → เลื่อนดู Test Data ที่ได้ | — |
| 4 | **ชี้ที่ Test Data ของ Pass Path (TC-P01)** | "Pass Path Data — SM2Input จริง: score=100, consecutive=0, EF=2.5 — คาดว่า interval=1" |
| 5 | **ชี้ที่ Test Data ของ Boundary Case (TC-E01, TC-E02)** | "Boundary Data — score=60 pass ได้ ส่วน score=59 fail ทันที — ทดสอบเส้นแบ่งชัดเจน" |
| 6 | **ชี้ที่คอลัมน์ Expected Output** | "Expected Output ระบุชัดว่าควรได้ผลอะไร — ทำให้เช็ค Pass/Fail ได้เลย" |
| 7 | หยุดนิ่ง 2 วินาที | "Test Data ที่ดีทำให้เราไม่ต้องเดาว่า test ผ่านหรือไม่ผ่าน — มีเกณฑ์ชัดเจนครับ" |
